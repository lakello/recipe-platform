КОНТЕКСТ ПРОЕКТА ДЛЯ НЕЙРОСЕТИ

Проект: DevOps pet-проект advanced-уровня.
Название: Платформа для рецептов и планирования питания.
Возможное англоязычное название: RecipeMeal Planner / Recipe Platform.

ОБЩАЯ ЦЕЛЬ ПРОЕКТА

Цель проекта — реализовать полноценное приложение с использованием современного DevOps-стека и научиться работать с каждой технологией на практике.

Проект не должен использовать технологии «для галочки». Каждая технология должна выполнять реальную задачу в архитектуре, разработке, доставке, эксплуатации или масштабировании приложения.

Приложение должно быть production-like, то есть максимально приближенным к реальному промышленному проекту: с инфраструктурой в облаке, CI/CD, Kubernetes, мониторингом, логированием, безопасностью, backup/restore, несколькими окружениями и документацией.

ОСНОВНОЕ ОПИСАНИЕ ПРОДУКТА

Платформа для рецептов и планирования питания — это сервис, где пользователи публикуют рецепты, планируют меню на неделю и автоматически формируют список покупок.

Платформа должна быть доступна через:

- Web-приложение
- Android-приложение
- Desktop-приложение для Windows
- Desktop-приложение для Linux

Целевая аудитория:

1. Обычные пользователи:
   - ищут рецепты
   - планируют питание
   - составляют список покупок
   - сохраняют любимые блюда
   - публикуют собственные рецепты

2. Авторы рецептов:
   - регулярно публикуют рецепты
   - получают подписчиков
   - взаимодействуют с аудиторией через комментарии

3. Администраторы и модераторы:
   - проверяют жалобы
   - модерируют рецепты, комментарии и пользователей
   - управляют категориями
   - блокируют нарушителей

ФУНКЦИОНАЛЬНОСТЬ ПРОЕКТА

1. Регистрация и авторизация

Система поддерживает несколько способов входа:

- Email + password
- Google OAuth
- Яндекс OAuth

При регистрации по email пользователь вводит:
- email
- пароль
- имя пользователя

Система должна:
- проверить уникальность email
- проверить сложность пароля
- сохранить пользователя
- отправить письмо подтверждения email
- выдать access token и refresh token после подтверждения или успешного входа

При авторизации по email пользователь вводит:
- email
- пароль

Система должна:
- проверить email и пароль
- выдать JWT access token
- выдать refresh token
- сохранить информацию о сессии

OAuth через Google и Яндекс:
- пользователь перенаправляется на страницу Google или Яндекс
- приложение получает OAuth callback
- создаёт пользователя, если его ещё нет
- привязывает OAuth-аккаунт к существующему пользователю, если email совпадает
- выдаёт собственные JWT-токены приложения

JWT — JSON Web Token.
Это строка-токен, которую сервер выдаёт пользователю после входа.
Пользователь отправляет JWT в каждом запросе к API в заголовке:

Authorization: Bearer <token>

JWT позволяет серверу понять:
- кто сделал запрос
- какие права у пользователя
- когда токен истекает

В проекте используются:
- Access token — короткоживущий токен, например 15 минут
- Refresh token — долгоживущий токен, например 7–30 дней, используется для получения нового access token

11. Автоматический список покупок

Система формирует список покупок на основе плана питания.

Алгоритм:
1. Пользователь добавляет рецепты в недельный план.
2. Система получает ингредиенты всех рецептов.
3. Система пересчитывает количество ингредиентов с учётом порций.
4. Система объединяет одинаковые ингредиенты.
5. Система группирует продукты по категориям.
6. Пользователь получает список покупок.

Пример:

Овощи:
- Картофель — 2 кг
- Морковь — 500 г

Мясо:
- Куриное филе — 1.5 кг

Молочные продукты:
- Молоко — 2 л

Пользователь может:
- отметить продукт как купленный
- добавить продукт вручную
- удалить продукт
- изменить количество
- экспортировать список

12. Публичные и приватные рецепты

Рецепт может быть:
- публичным
- приватным

Приватный рецепт доступен только автору.

Черновик доступен только автору.
Скрытый рецепт недоступен публично, но доступен модераторам и администраторам.

При помощи в разработке этого проекта нужно учитывать:

1. Это DevOps pet-проект advanced-уровня.
2. Нельзя предлагать технологии без реальной роли в проекте.

4. Предпочтительно использовать Yandex Cloud.
5. Kubernetes — Yandex Managed Kubernetes.
6. CI/CD — GitHub Actions.
7. Docker images — GitHub Container Registry.
8. Backend — FastAPI, если пользователь не выберет другое.
9. Frontend — React + TypeScript.
10. Android — Kotlin + Jetpack Compose.
11. Desktop — Tauri + React.
12. Инфраструктура описывается Terraform.
13. Linux VM настраиваются Ansible.
14. Секреты не должны попадать в Git.
15. Для деплоя использовать Helm.
16. Для входящего трафика использовать Nginx Ingress Controller.
17. Для HTTPS использовать cert-manager и Let’s Encrypt.
18. Для мониторинга использовать Prometheus/Grafana.
19. Для логов использовать Loki.
20. Для трассировки использовать OpenTelemetry + Tempo или Jaeger.
21. Нужно поддерживать local/dev/staging/production окружения.
22. Нужно объяснять не только «что сделать», но и «зачем это нужно».
23. Нужно помогать писать код, конфиги, Terraform, Ansible, Helm, Dockerfile, GitHub Actions и документацию.
24. Если есть несколько вариантов решения, желательно предлагать оптимальный и объяснять компромиссы.
